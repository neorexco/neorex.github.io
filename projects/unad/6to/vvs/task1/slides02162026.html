1. Verificación vs. Validación

Verificación

"¿Estamos construyendo el producto correctamente?"

Revisar que el software cumpla los requisitos técnicos y el diseño inicial. Es una revisión interna (código, especificaciones).

Validación

"¿Estamos construyendo el producto correcto?"

Asegurar que el software le sirva al usuario y solucione su problema real. Es útil, claro y adecuado.

Ejemplo (App de Gastos):
Verificar: El sistema guarda datos y calcula restas sin errores.
Validar: El usuario siente que la app le ayuda a ahorrar y entiende la interfaz.

2. ¿Qué es un Caso de Prueba?

Es un conjunto de condiciones o variables bajo las cuales un evaluador determinará si una aplicación satisface los requerimientos o funciona correctamente.

5 Elementos indispensables:

  • ID del Caso: Identificador único (ej. CP-001).
  • Descripción/Objetivo: Qué funcionalidad se va a probar.
  • Pre-condiciones: Qué debe estar listo antes de probar (ej. "Usuario logueado").
  • Pasos a ejecutar: Instrucciones paso a paso de qué hacer.
  • Resultado Esperado: Qué debería pasar si todo funciona bien (ej. "Aparece mensaje de éxito").

3. Tipos de Pruebas de Software

Existen diversas formas de clasificar las pruebas, las más comunes son:

Por Nivel (La Pirámide):
  • Unitarias: Prueban la parte más pequeña del código (funciones).
  • Integración: Prueban cómo interactúan módulos entre sí.
  • Sistema: Prueba el software completo.
  • Aceptación: El usuario final aprueba el software.
Por Objetivo:
  • Funcionales: ¿Qué hace el sistema? (Login, compras).
  • No Funcionales: ¿Cómo lo hace? (Rendimiento, Seguridad, Usabilidad).

4. Automatización de Pruebas

Consiste en usar software para controlar la ejecución de pruebas, comparando los resultados obtenidos con los esperados. Su fin es reducir esfuerzo manual, costos y tiempo.

Herramienta: Katalon Studio

  • Versátil: Pruebas API, Web, Móvil y Escritorio.
  • Para todos: Permite grabar acciones (sin código) o usar Scripts (con código).
  • Integración: Compatible con CI/CD, Git y reportes detallados.
  • Ayuda a mejorar la colaboración entre QA y Desarrollo.

5. Pruebas de Regresión

Garantizan que el software siga funcionando tras un cambio (parches, mejoras o configuraciones). Evitan que "arreglar algo rompa otra cosa".

¿Por qué son importantes?

  • Detección de efectos secundarios: Encuentra conflictos entre código nuevo y viejo.
  • Estabilidad: Asegura que las funciones críticas no se degraden.
  • Ahorro: Detectar errores temprano es más barato que corregirlos en producción.
  • Confianza: Mantiene la experiencia del usuario consistente.
